Self-Host

Run Your Own Relay Server

The VaultMesh relay is a lightweight, stateless WebSocket forwarder that helps devices reach each other across NAT. It never decrypts your data — it only forwards encrypted payloads between authenticated devices. Deploy it anywhere in minutes.

What Is the Relay?

VaultMesh devices sync directly over your local network (LAN) using libp2p mDNS — no relay needed. When devices are on different networks (e.g., home vs. office), the relay provides a rendezvous point for NAT traversal.

Privacy guarantee: The relay operates on protocol version X-VaultMesh-Protocol: 2 and is intentionally stateless — it authenticates device sessions but never buffers, stores, or reads vault payloads. Running your own relay means even the relay operator is you.

You need a relay only if your devices are:

  • On different networks (home, office, mobile data)
  • Behind double-NAT or CGNAT
  • You want sync to work without being on the same Wi-Fi

System Requirements

The relay is a single compiled Rust binary with minimal dependencies.

ComponentRequirement
OSLinux (x86_64 / arm64), macOS, or Windows Server
RAM64 MB minimum (no state is kept per connection)
Disk< 20 MB for the binary
NetworkStatic IP or domain name, inbound TCP port (default: 8080)
TLSRecommended — use a reverse proxy (nginx / Caddy) or provide cert paths

Option 1 — Single Binary

Download the pre-compiled relay binary for your platform and run it directly.

Linux x86_64 Linux arm64 macOS

Start the relay:

# Make executable and start (Linux/macOS)
chmod +x vault-relay
./vault-relay --port 8080 --log-level info

# With TLS certificates
./vault-relay \
  --port 443 \
  --tls-cert /etc/ssl/certs/your.crt \
  --tls-key  /etc/ssl/private/your.key

Option 2 — Docker

Pull the official image from GitHub Container Registry and run it as a container.

# Pull the latest image
docker pull ghcr.io/freehuoshan/vault-relay:latest

# Run with default settings (port 8080)
docker run -d \
  --name vault-relay \
  -p 8080:8080 \
  ghcr.io/freehuoshan/vault-relay:latest

# Run with custom environment variables
docker run -d \
  --name vault-relay \
  -p 8080:8080 \
  -e RELAY_PORT=8080 \
  -e RELAY_LOG_LEVEL=info \
  -e RELAY_MAX_CONNECTIONS=500 \
  ghcr.io/freehuoshan/vault-relay:latest

Docker Compose

For a production setup with automatic TLS via Caddy:

# docker-compose.yml
services:
  relay:
    image: ghcr.io/freehuoshan/vault-relay:latest
    restart: unless-stopped
    environment:
      RELAY_PORT: "8080"
      RELAY_LOG_LEVEL: "info"
    expose:
      - "8080"

  caddy:
    image: caddy:2-alpine
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - caddy_data:/data
    depends_on:
      - relay

volumes:
  caddy_data:
# Caddyfile
relay.yourdomain.com {
  reverse_proxy relay:8080
}

Option 3 — Build from Source

# Requires Rust toolchain (rustup.rs)
git clone https://github.com/freehuoshan/vaultmesh.git
cd vaultmesh
cargo build --release -p vault-relay

# Binary will be at:
./target/release/vault-relay

Configuration Reference

All options can be set via CLI flags or environment variables.

FlagEnv VariableDefaultDescription
--port RELAY_PORT 8080 TCP port to listen on
--bind RELAY_BIND 0.0.0.0 Interface address to bind
--tls-cert RELAY_TLS_CERT — Path to PEM certificate file
--tls-key RELAY_TLS_KEY — Path to PEM private key file
--max-connections RELAY_MAX_CONNECTIONS 1000 Maximum concurrent device connections
--session-ttl RELAY_SESSION_TTL 3600 Inactive session timeout in seconds
--log-level RELAY_LOG_LEVEL info error / warn / info / debug

Connect VaultMesh Clients

Once your relay is running, configure it in the VaultMesh desktop app:

  1. Open Settings → Sync → Relay in the VaultMesh desktop app.
  2. Enter your relay URL, e.g. wss://relay.yourdomain.com.
  3. Click Save. All devices sharing your vault should use the same URL.

WSS required in production. VaultMesh clients enforce wss:// (TLS) for relay connections outside of localhost. Use a reverse proxy to terminate TLS if you are not providing cert paths directly to the binary.

Systemd Service (Linux)

Run the relay as a system service that starts on boot:

# /etc/systemd/system/vault-relay.service
[Unit]
Description=VaultMesh Relay Server
After=network.target

[Service]
Type=simple
User=vault-relay
ExecStart=/usr/local/bin/vault-relay --port 8080 --log-level info
Restart=on-failure
RestartSec=5
NoNewPrivileges=true
PrivateTmp=true

[Install]
WantedBy=multi-user.target
sudo systemctl daemon-reload
sudo systemctl enable --now vault-relay
sudo systemctl status vault-relay

Health Check & Monitoring

The relay exposes an HTTP health endpoint:

GET /health

# Example response
{
  "status": "ok",
  "version": "1.0.0",
  "connections": 3,
  "uptime_seconds": 86400
}

Use this endpoint with your uptime monitor (UptimeRobot, Prometheus, etc.) to alert on relay downtime.

Note: The relay only forwards encrypted data — it is not a replacement for a VPN or a general-purpose proxy. Exposing a relay does not expose your vault contents. That said, restrict access if you prefer to limit which IP ranges can connect.

Upgrading

The relay is backwards-compatible within the same major protocol version. To upgrade, replace the binary (or pull a new Docker image) and restart the service. No database migrations or persistent state to worry about. 

# Docker upgrade
docker pull ghcr.io/freehuoshan/vault-relay:latest
docker compose up -d --force-recreate relay
